# Feature-Family Ablation and Threshold Sensitivity

This document specifies the ablation experiment that measures **how much each
feature family contributes** to the Stage-2 AI-authorship detector, and how
sensitive the detector's reported confusion matrix is to the decision threshold.
It is the design record for `scripts/run_family_ablation.py`.

For the module internals see [README.md](README.md); for the production training
pipeline see [training.md](training.md); for how the corpus is assembled see
[dataset_construction.md](dataset_construction.md).

> **This document holds the method, not the numbers.** Results are generated by
> the script into `ablation_report.md` alongside the CSVs (see §6). Numbers are
> reproduced here only where they justify a design decision, and each is cited to
> the run that produced it.

## 1. The questions

The detector uses 16 features in three families of unequal size:

| Family | Features | Count | Cost to compute |
|---|---|---:|---|
| Perplexity | F1–F6 | 6 | **High** — requires a forward pass through CodeGen-350M |
| Stylometric | F7–F12 | 6 | Low — AST walk |
| Distribution | F13–F16 | 4 | Low — token counting |

Two questions follow, and they are not the same question:

1. **Is each family necessary?** If removing a family costs little AUROC, it is
   redundant given the others.
2. **Is each family sufficient?** If a family alone reaches most of the full
   model's AUROC, it carries the signal largely on its own.

A family can be **sufficient but not necessary** when families are redundant —
two families encoding the same underlying signal will each do well alone while
neither looks costly to remove. Reading only `drop_*` arms would conclude
everything is dispensable; reading only `only_*` arms would conclude everything
is essential. The experiment runs both and they must be read together.

The practical stake is Phase B cost. Perplexity features are the *only* reason
feature extraction needs a GPU at all (see
[training.md](training.md#phase-b--extract-features-on-a-gpu)). If F1–F6 turn out
to be neither necessary nor sufficient, extraction collapses from a 20–40 minute
GPU job to a fast CPU pass, and the whole pipeline gets cheaper by an order of
magnitude. That is a result worth measuring properly rather than assuming.

## 2. Why the ablation runs on a single corpus

**Decision: `droidcollection` only, not the pooled corpus.**

Pooling corpora confounds feature importance with corpus identity. A feature that
separates CodeNet from DroidCollection will look like a strong AI detector in a
pooled run, because the corpora are not class-balanced against each other —
CodeNet is human-only. The model can score well by learning *which corpus a
sample came from*, and SHAP will duly report that corpus marker as the top
feature.

This is not hypothetical. `f7_avg_identifier_length` behaves exactly this way
across the three corpus configurations already run:

| Feature | Pooled (+CodeNet)<br/>`2026-07-17T13-16-24Z_gemini` | No CodeNet<br/>`2026-07-19T08-53-52Z_no-codenet` | Droid only<br/>`2026-07-19T15-43-36Z_droid-only-*` |
|---|---|---|---|
| `f7_avg_identifier_length` | **0.925 (rank 1)** | 0.186 (rank 6) | 0.285 (rank 5) |
| `f9_whitespace_consistency` | 0.418 (rank 2) | **0.742 (rank 1)** | **0.902 (rank 1)** |
| `f10_comment_to_code_ratio` | 0.416 (rank 3) | 0.637 (rank 3) | 0.767 (rank 2) |

*(mean |SHAP|, `headline.shap_mean_abs` in each run's `manifest.json`.)*

F7's importance **collapses five-fold** the moment CodeNet leaves the pool, while
the features that survive as top-ranked barely move. CodeNet is competitive
programming submissions, where identifiers are `n`, `i`, `dp`. The pooled model
had learned "short identifiers ⇒ human" — which is true of *CodeNet*, not of
human code. Ablating families on that corpus would measure the stylometric
family's ability to detect CodeNet, and report it as AI detection.

The same three runs show the downstream effect: leave-one-model-out recall on the
held-out `gemini` generator was **0.179** pooled and **0.547** with CodeNet
removed — the pooled detector was substantially blind to an unseen generator
because it had a corpus shortcut available instead.

**DroidCollection is chosen** as the single corpus because it is the only source
carrying both classes in volume (148k human / 98k AI Python rows) from a shared
distribution, and it spans 45 generators, so within-corpus generator diversity is
preserved.

**Known limit of this choice.** Single-corpus results are the cleanest available
measurement of family contribution, but they are *within-corpus*. The two
transfer runs (`transfer-droid-to-aig`, `transfer-aig-to-droid`) put cross-corpus
AUROC at **0.581** and **0.591** — near chance. Whichever family this ablation
credits, that credit is demonstrated within DroidCollection and should not be
assumed to survive a corpus change. §8 says what would.

## 3. Fixed hyperparameters across every arm

**Decision: one parameter set, reused by all seven arms, no per-arm tuning.**

Re-tuning each arm would mix two effects that must stay separate: the effect of
removing features, and the variance of `RandomizedSearchCV` sampling 50
candidates from a grid. An arm could then "beat" the full model purely by drawing
a luckier hyperparameter sample, and the delta would be uninterpretable.

The parameters are lifted verbatim from the tuned full-feature droid-only run,
`data/runs/2026-07-19T15-43-36Z_droid-only-qwen-qwen2-5-coder-32b-instruct/manifest.json`
(`headline.best_params`):

```python
subsample=0.9, n_estimators=300, min_child_weight=1,
max_depth=7, learning_rate=0.05, colsample_bytree=0.7
```

Note this is the **`headline`** block, not `shipped`. The two differ in that run
(`shipped` tuned to `n_estimators=200, min_child_weight=5, max_depth=10,
colsample_bytree=0.9`) because `ship()` re-runs the search over the full dataset.
The headline block is the correct source: it is the configuration whose published
metrics the ablation is measuring deltas against.

**The bias this introduces, stated plainly.** These parameters were tuned for 16
features. Reduced arms — especially `only_distribution` at 4 features — are
therefore evaluated under a configuration not tuned for them, and
`colsample_bytree=0.7` on 4 features samples under 3 columns per tree. Each
`only_*` arm's score is best read as a **lower bound** on what that family could
achieve with its own tuning. The alternative (per-arm tuning) trades this known,
directional bias for an unknown, non-directional one, which is the worse deal
when the goal is comparing arms to each other.

## 4. Five seeds, resampled end to end

**Decision: 5 seeds (42–46); each seed re-draws the rebalance *and* the split.**

The seed is threaded through three places — the majority-class downsample, both
`train_test_split` calls, and the XGBoost `random_state`. So the spread across
seeds captures **split variance and sampling variance**, not just model
initialization noise. Reporting only model-init variance would understate the
true uncertainty considerably, since which rows land in a 1,480-row test set
matters far more than how the trees initialize.

Deltas are **paired by seed**: each arm is compared against `full_16` fit on the
*same* seed, so the shared split variance cancels rather than being added twice.
The reported 95% CI is `1.96 × std(paired deltas) / √5`.

> **Five seeds is few.** It is enough to distinguish a family that costs several
> AUROC points from one that costs none, which is the decision the experiment
> exists to inform. It is *not* enough to resolve differences on the order of the
> CI itself. Treat overlapping intervals as "not separated by this experiment,"
> never as "shown equal."

## 5. Threshold selection at a fixed false-positive rate

**Decision: the operating threshold is chosen on the validation split at a target
FPR (default 5%), never on test.**

Two things are wrong with reporting metrics only at 0.5. First, 0.5 is a
reporting convention inherited from the classifier's default, not a choice
anybody made for this problem. Second, the errors are not symmetric: **flagging
human-authored code as AI is the expensive error.** A false positive puts a
developer's own work under suspicion; a false negative merely misses one AI
commit in a time series that aggregates thousands.

So each arm additionally reports its operating point at a fixed FPR. The
threshold is the `1 − target_fpr` quantile of the validation negatives' scores —
the smallest cut leaving at most `target_fpr` of human samples above it.

Selecting it on **validation** and applying it to **test** is what makes the
reported FPR an honest estimate. A threshold chosen on the test set would be
fitted to that set's negatives, and the reported FPR would be optimistic by
construction. This is why `split()` carves out a validation slice at all — the
detector itself has no other use for one.

Table 3 additionally sweeps the threshold from 0.05 to 0.95 for the full model,
so the precision/recall trade is visible rather than asserted.

> **Downstream, the threshold barely matters.** Deltx consumes `ai_confidence` as
> a *continuous* signal at index [4] of the 15-D vector. Nothing in the pipeline
> binarizes it. The threshold affects the reported confusion matrix and the human
> reading of it — not what Stage 4 receives.

## 6. The seven arms

| Arm | Features | Reads as |
|---|---:|---|
| `full_16` | 16 | Baseline; every delta is measured against it |
| `drop_perplexity` | 10 | Cost of removing F1–F6 → **necessity** of perplexity |
| `drop_stylometric` | 10 | Cost of removing F7–F12 → necessity of stylometric |
| `drop_distribution` | 12 | Cost of removing F13–F16 → necessity of distribution |
| `only_perplexity` | 6 | How far F1–F6 get alone → **sufficiency** of perplexity |
| `only_stylometric` | 6 | How far F7–F12 get alone → sufficiency of stylometric |
| `only_distribution` | 4 | How far F13–F16 get alone → sufficiency of distribution |

7 arms × 5 seeds = **35 fits**, a few minutes on CPU.

**Families are unequal (6 / 6 / 4) and deltas are reported raw**, not normalized
per feature. Normalizing would imply features are interchangeable units, which
they are not — F3 (`sequence_perplexity`, mean |SHAP| 0.023) and F9
(`whitespace_consistency`, 0.902) differ by a factor of 39 in the droid-only run.
The consequence is that `only_distribution` is handicapped by having four
features rather than six, and its result should be read with that in mind. It is
disclosed rather than corrected away.

## 7. Running it

```bash
# Defaults: droidcollection only, 5% target FPR
poetry run python scripts/run_family_ablation.py

# Different operating point, or a different corpus slice
poetry run python scripts/run_family_ablation.py --target-fpr 0.01
poetry run python scripts/run_family_ablation.py --sources droidcollection aigcodeset
```

| Flag | Default | Purpose |
|---|---|---|
| `--features` | `data/processed/train_features.parquet` | Phase B feature matrix |
| `--sources` | `droidcollection` | `source_dataset` values to keep — see §2 before widening |
| `--target-fpr` | `0.05` | FPR the validation-selected threshold targets |
| `--out` | `data/runs/ablation_families` | Output directory |

Requires Phase B to have run; the script exits with a pointer if the parquet is
absent. It does **not** need a GPU — features already exist as numbers.

### Outputs — six files

| File | Contents |
|---|---|
| `ablation_report.md` | **The paper-ready artifact**: Tables 1–3 rendered as markdown |
| `ablation_summary.csv` | Per-arm aggregates: AUROC ± std, AUPRC, paired ΔAUROC, 95% CI |
| `ablation_raw.csv` | One row per (arm, seed) — every metric at both thresholds |
| `threshold_sweep_summary.csv` | Mean ± std per threshold, 0.05→0.95 |
| `threshold_sweep_raw.csv` | Per-seed sweep rows |
| `config.json` | Sources, seeds, fixed params, target FPR, family sizes |

> **Capture gap.** Unlike `train_detector.py`, this script does not write a
> provenance manifest — no git commit, no `dataset.sha256`, no package versions.
> `config.json` records the knobs but not the inputs or the code state. Results
> are reproducible from a clean tree at a known commit; they are not
> self-describing the way a `data/runs/` manifest is. Record the commit alongside
> the output directory until this is wired into `common/provenance.py`.

## 8. What this experiment does not answer

Stated so nobody over-reads the report:

- **Cross-corpus family contribution.** Everything here is within DroidCollection
  (§2). A family that carries the signal here may be the one that fails to
  transfer — indeed, the transfer runs' near-chance AUROC means *something*
  fails to transfer, and this design cannot say which family. Answering it needs
  the arms re-run under a train-on-A / test-on-B protocol.
- **Per-feature attribution.** Arms are family-granular. For individual features,
  read `headline.shap_mean_abs` from a training run's manifest — but note SHAP
  reports what the model *used*, which given correlated features is not the same
  as what the model *needed*. That difference is precisely why this ablation
  exists as a separate experiment rather than being read off the SHAP ranking.
- **Whether a reduced model should ship.** A cheap arm scoring within noise of
  `full_16` is a candidate, not a decision — it would still need its own tuning
  (§3), its own LOMO evaluation, and a transfer check.
- **Interactions between families.** Seven arms cover full, each-dropped, and
  each-alone. Pairwise arms (`perplexity + distribution`, etc.) are not run; the
  design assumes families are the meaningful unit.
